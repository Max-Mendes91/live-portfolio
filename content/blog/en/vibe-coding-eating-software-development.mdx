# Vibe Coding Is Eating Software Development — And Not Everyone Is Happy

Andrej Karpathy coins a term in February 2025. By December it's Collins Word of the Year. By March 2026, there's an academic workshop about it and senior engineers are posting horror stories about "development hell." That's a fast arc.

Here's my take: vibe coding is real, it's useful, and the people most upset about it are upset for the wrong reasons.

## What Vibe Coding Actually Is

The original definition from Karpathy is blunt: you describe what you want, the AI generates code, and you "forget that the code even exists." You accept output without reviewing it, nudge it with follow-up prompts, and ship when it works.

That's not how I use AI when coding. I review every important piece. I test it. I understand the structure before it goes anywhere near production. Simon Willison put it well: if you've reviewed and understood the code, that's using an LLM as a typing assistant, not vibe coding.

But plenty of people are using the Karpathy definition literally. Non-coders building apps. Founders shipping MVPs without a technical hire. Solo developers building internal tools faster than any team could. That's the part that's making senior engineers nervous.

## The Criticism Is Partly Right

The concerns aren't imaginary. The State of Software Delivery 2025 report found that most developers now spend more time debugging AI-generated code than before, not less. Security firms keep finding the same fingerprints: misconfigured Firebase databases, missing Supabase Row Level Security, hardcoded API keys. GitClear analyzed 211 million lines of code from 2020-2024 and found AI-assisted code is harder to maintain long-term.

That's real. Someone who vibe-coded a customer-facing app with no understanding of the security model has created a liability, not a product. And as more of these apps hit production, someone will have to clean it up.

## The Criticism Is Also Missing the Point

The loudest complaints are framed as "vibe coding is bad for software." But what I keep reading between the lines is something else: "people who don't know what I know are now building things."

That's not a technical argument. That's a guild protecting its territory.

I've been building websites and automation systems for local Polish businesses. Small restaurants, nail salons, barbers. None of them need a team of senior engineers. They need a working website with a contact form and decent SEO. I use AI-assisted coding to build those faster and cheaper than I could otherwise. That's not a crisis. That's the market working.

The crisis is real for companies that have vibe-coded their way into 50,000 lines of AI-generated spaghetti and now need to add a feature. The solution there isn't "ban AI from development." It's "understand what you're building."

## What Actually Matters in 2026

I'll take a stance here: the developers who are scared of vibe coding are the ones whose value was in knowing syntax and APIs. That was always a fragile position. The developers who are fine are the ones whose value is in knowing what to build, why to build it, and whether what got built is correct.

Vibe coding raises the floor. Anyone can now produce working code for simple problems. That's genuinely good. The ceiling, understanding systems, making architectural decisions, spotting the security hole the AI didn't flag, hasn't moved. If anything, it got more valuable, because someone has to supervise all these AI-generated apps.

I'm using Claude via [AI integration work](/en/services/ai-integration) on almost every project now. I prompt, I review, I modify. I don't fully give in to the vibes. But I also don't pretend I hand-write everything from scratch to preserve some purity that stopped mattering two years ago.

The developers who'll struggle aren't the ones who adopt AI coding. They're the ones who refuse it, and the ones who adopt it without judgment. The middle position, where you use the tools and stay responsible for the output, is where the actual work happens.

Vibe coding is eating software development. Some of that is messy and some of it will cause problems. But the core shift, that describing what you want in plain language is now a valid way to build software, is here and it's not reversing.
